## Overview

This document provides a comprehensive reference for all parameters, constants, and configurable values in the Kach protocol. Parameters are organized by module and indicate whether they are governance-controlled or hardcoded constants.

## Governance Parameters

These parameters can be adjusted by governance (currently admin multisig):

### Protocol-Level Parameters

**Module**: `governance.move`

| Parameter | Current Value | Unit | Description |
|-----------|--------------|------|-------------|
| `protocol_fee_bps` | 700 | bps | Protocol fee on gross interest (7%) |
| `max_utilization_bps` | 8000 | bps | Maximum pool utilization (80%) |
| `min_lock_duration_seconds` | deployment-specific | seconds | Minimum position lock-up period |
| `max_lock_duration_seconds` | deployment-specific | seconds | Maximum position lock-up period |
| `min_trust_score_threshold` | 60 | score | Minimum trust score to draw credit (standard draws) |
| `default_tenor_seconds` | varies | seconds | Default loan duration (7-90 days) |
| `global_pause` | false | boolean | Emergency pause flag |

**Adjustable By**: Admin multisig
**Future**: Token governance

### Pool Parameters

**Module**: `pool.move`

| Parameter | Current Value | Unit | Description |
|-----------|--------------|------|-------------|
| `senior_target_bps` | 5000 | bps | Senior tranche target allocation (50%) |
| `junior_target_bps` | 5000 | bps | Junior tranche target allocation (50%) |
| `protocol_reserve_target_bps` | collects protocol fees | bps | Reserve grows from protocol fee; scales with pool size |

**Note**: Per-pool parameters can vary by asset type.

### Credit Line Parameters

**Module**: `credit_engine.move`

Set per Attestator during credit line creation:

| Parameter | Typical Range | Unit | Description |
|-----------|--------------|------|-------------|
| `max_outstanding` | $100K-$5M | USD | Maximum Attestator limit |
| `default_tenor_seconds` | 604800-7776000 | seconds | Default loan duration |
| `default_interest_rate_bps` | 20 bps (standard draws), 35-320 (prefund tenors) | bps | Default interest rate |

**Adjustable By**: Admin when creating/modifying credit lines

### Tranche Constants

**Module**: `tranche.move`

#### Yield Distribution
| Tranche | Share of Interest | Risk Level |
|---------|------------------|------------|
| Protocol Fee | 7% | N/A |
| Junior | Capital-weighted share using multiplier (e.g., 1.3×) | Highest |
| Senior | Capital-weighted share using multiplier (e.g., 0.8×) | Lowest |

**Distribution Method**:
- Compute weights: `w_senior = senior_deposits × senior_multiplier`, `w_junior = junior_deposits × junior_multiplier`
- Total weight: `w_total = w_senior + w_junior`
- Senior share = `(interest_after_fee × w_senior) / w_total`
- Junior share = `(interest_after_fee × w_junior) / w_total`
**Loss Waterfall**:
- Protocol reserve absorbs first
- Junior tranche absorbs losses next until depleted
- Senior tranche absorbs remaining loss

## Parameter Change Process

### Current Process (Phase 0)

1. **Proposal**: Admin reviews need for parameter change
2. **Analysis**: Risk team evaluates impact
3. **Decision**: Admin multisig (4-of-6) approves
4. **Execution**: Admin calls governance update function
5. **Event**: `ParameterUpdated` event emitted
6. **Communication**: Change announced to community

**Multisig Composition**:
- Bayta CEO
- Risk lead
- Engineering lead
- 2 independent advisors
- Aptos Foundation observer

### Future Process (Planned)

**Phase 1 (Guarded Governance)**:
- Add time-locks (e.g., 48-hour delay)
- Add observers with veto rights
- Require published rationale
- Community feedback period

**Phase 2 (Token Governance)**:
- KACH token holders vote on proposals
- Governance committee elected by token holders
- Parameter changes require token vote
- Emergency changes still possible by multisig

**Phase 3 (Full Decentralization)**:
- Token governance for all parameters
- Delegated authority for routine changes
- Emergency multisig only for critical security

## Parameter Ranges and Constraints

### Protocol Fee

**Current**: 700 bps (7%)
**Range**: 0-1000 bps (0-10%)
**Rationale**: Balance protocol sustainability with LP competitiveness

**Impact of Changes**:
- Higher → More reserve, less LP yield (reserve grows via fee capture)
- Lower → Less reserve, more LP yield
- Recommended: 5-7% based on market conditions and runway needs

### Max Utilization

**Current**: 8000 bps (80%)
**Range**: 5000-9000 bps (50-90%)
**Rationale**: Balance yield opportunity with withdrawal liquidity

**Impact of Changes**:
- Higher → More yield, less withdrawal buffer
- Lower → More buffer, less yield potential
- Recommended: 70-85% based on LP liquidity needs

### Trust Score Thresholds

**Current**: 60 (standard), 95 (prefund)
**Range**: 50-70 (standard), 90-98 (prefund)
**Rationale**: Balance credit access with default risk

**Impact of Changes**:
- Lower standard → More access, higher risk
- Higher standard → Less access, lower risk
- Prefund should remain very high (95+)

### Interest Rates

**Current**: 35-320 bps depending on tenor
**Adjustment Method**: Contract upgrade (hardcoded)
**Future**: May make governance-adjustable

**Considerations**:
- Market rates (DeFi and traditional)
- Default risk premiums
- Operational costs
- Competitive positioning
- LP yield targets

## Monitoring Parameter Health

### Key Metrics to Track

**Utilization**:
- Target: 70-80%
- Warning: &lt;50% or $gt;85%
- Action: Adjust rates or marketing

**Default Rate**:
- Target: $lt;2% annually
- Warning: >3%
- Action: Tighten trust score requirements

**Protocol Reserve**:
- Target: 5-10% of TVL
- Warning: $lt;3% of TVL
- Action: Increase protocol fee temporarily

**LP Yields**:
- Target: 12-20% APY
- Warning: $lt;8% or >25%
- Action: Adjust rates or utilization target

**Trust Score Distribution**:
- Target: 70% of borrowers >80 score
- Warning: >30% of borrowers 60-70 range
- Action: Review credit quality
